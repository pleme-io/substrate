# Adding a Leptos Web Application

> **Note:** This is one of 6 scaffold types available in substrate. See
> [scaffolds.md](scaffolds.md) for the full list: Leptos PWA, Rust Service,
> Rust CLI Tool, Dioxus Desktop, GPU Application, and Ruby Gem.

## Scaffolding a New App

The fastest way to create a new Leptos PWA:

```nix
# In nix repl or a script:
scaffold = import "${substrate}/lib/leptos-app-scaffold.nix" { inherit lib; };

# Standard web app with auth, PWA, i18n
app = scaffold.generate ({
  name = "my-product";
  displayName = "My Product";
  description = "A new pleme-io product";
  primaryColor = "#dc143c";
  locale = "pt-BR";
} // scaffold.templates.standard);

# app.files — attrset of path -> content (write to disk)
# app.meta — app metadata
# app.deployment — deployment spec for substrate archetypes
```

### Templates

| Template | Features |
|----------|----------|
| `minimal` | Routing + theme only |
| `standard` | Auth + PWA + i18n + observability |
| `product` | Standard + admin + search + payments |
| `internal` | Auth + admin + observability (no PWA) |

## Building from Source

Build Leptos web applications with dual-target compilation: a native SSR server
binary and a client-side WASM bundle with hydration.

## Quick Start

### Consumer `flake.nix`

```nix
{
  inputs = {
    nixpkgs.url = "github:nixos/nixpkgs?ref=nixos-unstable";
    fenix = {
      url = "github:nix-community/fenix";
      inputs.nixpkgs.follows = "nixpkgs";
    };
    substrate = {
      url = "github:pleme-io/substrate";
      inputs.nixpkgs.follows = "nixpkgs";
    };
  };

  outputs = { self, nixpkgs, fenix, substrate, ... }:
    (import "${substrate}/lib/leptos-build-flake.nix" {
      inherit nixpkgs substrate;
    }) {
      inherit self;
      name = "my-leptos-app";
    };
}
```

### What You Get

| Output | Description |
|--------|-------------|
| `packages.default` | Combined SSR binary + CSR WASM bundle |
| `packages.ssrBinary` | Native SSR server binary |
| `packages.csrBundle` | Optimized WASM + JS + assets |
| `packages.dockerImage` | Docker image with SSR binary |
| `devShells.default` | Dev shell with all tooling |
| `apps.default` | Run the SSR server directly |
| `overlays.default` | Nixpkgs overlay |

## How It Works

The builder compiles your Leptos crate twice:

1. **SSR binary** (native target): `cargo build --release --features ssr`
2. **CSR bundle** (wasm32-unknown-unknown): `cargo build --release --target wasm32-unknown-unknown --features hydrate` + `wasm-bindgen` + `wasm-opt`

The combined output places the SSR binary in `/bin/` and the CSR bundle in
`/static/`. The SSR binary serves the WASM bundle from the static directory.

## Configuration

### Feature Flags

The builder uses Cargo features to separate SSR and CSR code:

```toml
# Cargo.toml
[features]
ssr = ["leptos/ssr", "leptos_router/ssr"]
hydrate = ["leptos/hydrate", "leptos_router/hydrate"]
```

Override with `ssrFeatures` and `csrFeatures` parameters:

```nix
(import "${substrate}/lib/leptos-build-flake.nix" { ... }) {
  name = "my-app";
  ssrFeatures = "ssr,my-custom-feature";
  csrFeatures = "hydrate";
};
```

### Static Assets

Place static assets in a `public/` directory. Pass it to the builder:

```nix
{
  name = "my-app";
  staticAssets = ./public;  # copied to /static/ alongside WASM bundle
  indexHtml = ./index.html;  # if not generated by Leptos
}
```

### Docker Configuration

```nix
{
  name = "my-app";
  tag = "0.1.0";
  port = 3000;        # SSR server port
  healthPort = 3001;  # Health check port
}
```

### CSR-Only Deployment (Hanabi)

For pure client-side rendering without SSR, use `mkLeptosDockerImageWithHanabi`
directly via `lib.${system}`:

```nix
substrateLib = substrate.lib.${system};
csrImage = substrateLib.mkLeptosDockerImageWithHanabi {
  name = "my-app";
  csrBundle = (substrateLib.mkLeptosBuild { name = "my-app"; src = ./.; }).csrBundle;
  webServer = hanabi;  # Hanabi binary from crate2nix
};
```

## Development

The dev shell includes:

- Rust stable toolchain + `wasm32-unknown-unknown` target
- `trunk` for CSR dev server with hot reload
- `wasm-bindgen-cli` and `binaryen` (wasm-opt)
- `tailwindcss` for CSS processing
- `npm` and Node.js for `@material/web` and MUI island JS
- `cargo-watch` for auto-rebuild
- `rust-analyzer` for IDE support

### Dev Workflow

```bash
# Enter dev shell
nix develop

# CSR development with hot reload
trunk serve

# SSR development
cargo run --features ssr

# Production build
nix build
```

### Devenv Module

For projects using devenv.sh:

```nix
# devenv.nix
{ pkgs, ... }: {
  imports = [ "${substrate}/lib/devenv/leptos.nix" ];
}
```

## Project Structure

Recommended Leptos project layout:

```
my-leptos-app/
  Cargo.toml          # features: ssr, hydrate
  Cargo.lock
  flake.nix           # uses leptos-build-flake.nix
  Trunk.toml          # CSR dev server config
  tailwind.config.js  # Tailwind CSS config
  public/
    manifest.json     # PWA manifest
    sw.js             # Service worker
    icons/            # PWA icons
  js/
    material-web.js   # @material/web bundle
  src/
    main.rs           # mount_to_body for CSR
    lib.rs            # shared app code
    app.rs            # root component
```

## Integration with Substrate Archetypes

Deploy via the unified infrastructure theory:

```nix
# In your infrastructure declaration
mkHttpService {
  name = "my-leptos-app";
  image = "ghcr.io/pleme-io/my-leptos-app:latest";
  ports = [{ port = 3000; protocol = "http"; }];
  health = { path = "/healthz"; port = 3000; };
}
```

This renders to Kubernetes (via `mkMicroservice`), tatara (via JobSpec), and
WASI (via component config) simultaneously.
